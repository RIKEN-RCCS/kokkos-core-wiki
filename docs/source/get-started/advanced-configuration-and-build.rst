高度な構成とビルド
==================

nvcc_wrapper: なぜ存在するのか、そして何をするのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

一般に（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で構成される場合を除いて）、Kokkos に依存するパフォーマンスポータブルなコードは、``CMake`` において自身を ``CXX`` コードとして識別します。
これは、``CXX`` コードをコンパイルするために ``CMake`` が使用するコンパイラが、Kokkos で有効化されたバックエンドおよびアーキテクチャ向けにコードをコンパイルできなければならないことを意味します。

``nvcc`` を使用する ``CUDA`` バックエンドでは、これは構成時に ``CMAKE_CXX_COMPILER=nvcc`` を設定することを意味します。しかし ``nvcc`` では、ホストコンパイルとデバイスコンパイルに別々のフラグが必要です。
別々のフラグを必要とするこの ``nvcc`` の要件は、同じターゲットにリンクされる他のライブラリもこれに従う必要があることを意味します。
``clang`` のような他のコンパイラは、ホストフラグとデバイスフラグを区別する必要がないため、ラッパースクリプトを必要とせず、``CMAKE_CXX_COMPILER`` を介して直接使用できます。

この問題を解決するために、Kokkos には ``bin`` サブディレクトリにある ``nvcc_wrapper`` という小さな ``bash`` スクリプトが付属しています。
このスクリプトには2つの機能があります。コンパイルコマンドとリンクコマンドを ``nvcc`` にリダイレクトすること、そして与えられたコンパイラフラグとリンカフラグをホストコンパイラ用とデバイスコンパイラ用に振り分けることです。

これを明示的に使用するには、``CMAKE_CXX_COMPILER`` を適切に設定します。コンパイラがデバイスコードをコンパイルできない場合、Kokkos は ``kokkos_launch_compiler``（次のセクションを参照）を使用して、呼び出しを自動的に ``nvcc_wrapper`` にリダイレクトします。

kokkos_launch_compiler: 何をするのか、そしてどう制御するのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

複数のライブラリに依存する複雑なソフトウェアプロジェクトでは、有効化されたバックエンドのために Kokkos が必要とするコンパイラが、プロジェクトが依存する他のライブラリの1つをコンパイルできない場合があります。
このような状況に対処するため、Kokkos は ``kokkos_launch_compiler`` という別の ``bash`` スクリプトを導入しました。
このスクリプトは、Kokkos を使用する ``C++`` ファイルをコンパイルするコンパイラコマンドとリンカコマンド**のみ**を、Kokkos コードをコンパイルできるコンパイラ（例: ``CUDA`` バックエンド用の ``nvcc_wrapper``）にリダイレクトします。Kokkos を使用しない ``C++`` ファイル、または異なる言語のファイルのコンパイラコマンドとリンカコマンドはリダイレクトされません。

``bin`` サブディレクトリにあるこのスクリプトは、``CMake`` におけるコンパイラランチャーのように使用することを意図しています。
しかし（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で構成される場合を除いて）、Kokkos は ``CMake`` が使用する ``CXX`` コンパイラが、有効化されたバックエンド向けにコードをコンパイルできるかどうかを検出しようとします。``CXX`` コンパイラがバックエンドコードをコンパイル**できない**場合、Kokkos は自動的に ``kokkos_launch_compiler`` を使用します。``nvcc_wrapper`` にリダイレクトする場合、``nvcc`` のホストコンパイラを ``CXX`` コンパイラに設定します。これは、Kokkos のパフォーマンスポータブルなビルドシステムの使用を簡素化することを目的としています。

これはほとんどのユースケースをカバーしますが、Kokkos は ``kokkos_launch_compiler`` を**常に**使用する、または**決して**使用しないようにユーザーが要求する方法を提供しています。
``kokkos_launch_compiler`` を常に使用するには、``find_package`` を呼び出す際に ``launch_compiler`` コンポーネントを要求できます:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS launch_compiler)

対照的に、``kokkos_launch_compiler`` を決して使用したくない場合:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS separable_compilation)

バックエンドの CMake 言語でコンパイルする
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

対応する専用の ``CMake`` 言語を持つ ``CUDA`` および ``HIP`` バックエンドでは、Kokkos はそれぞれの ``CMake`` 言語でのみコンパイラフラグとリンカフラグを設定するように構成できます。これにより、Kokkos は ``CMake`` に対して（``CXX`` ライブラリとして識別される**代わりに**）``CUDA`` または ``HIP`` ライブラリとして識別されるようになります。これにより、追加のホストバックエンドを有効化できます（例: ``CUDA``、``OPENMP``、``SERIAL`` を ``CUDA`` 言語モードでコンパイルできます）。
これは、Kokkos にリンクするファイルもそれぞれの ``CMake`` 言語として識別され、Kokkos で有効化されたアーキテクチャ向けにコンパイルされる必要があることを意味します。
この動作を有効にするには、`Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE <configuration-guide.html#backend-specific-options>`_ オプションを有効にします。詳細な例と説明については、このオプションに関するノートを確認してください。

複数の言語でコンパイルする
^^^^^^^^^^^^^^^^^^^^^^^^^^

複雑なソフトウェアプロジェクトでは、Kokkos を使用する複数のライブラリを使用する場合があります。これらのうちの1つがバックエンドの ``CMake`` 言語（例: ``Cuda`` や ``HIP``）で Kokkos を使用し、他のライブラリが Kokkos を ``CXX`` ライブラリとして使用する場合、両方のモードでコンパイルできる Kokkos インストールが必要になります。
これを可能にするため、Kokkos は `Kokkos_ENABLE_MULTIPLE_CMAKE_LANGUAGES <configuration-guide.html#backend-specific-options>`_ オプションを提供しています。詳細と要件については、このオプションに関するノートを確認してください。
