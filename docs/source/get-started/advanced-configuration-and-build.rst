高度な構成とビルド
==================

nvcc_wrapper: なぜ必要で何をするのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

一般的に（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で構成される場合を除く）、Kokkos に依存するパフォーマンスポータブルなコードは、``CMake`` において自身を ``CXX`` コードとして識別します。
これは、``CMake`` が ``CXX`` コードのコンパイルに使用するコンパイラが、Kokkos で有効化されたバックエンドとアーキテクチャ向けにコードをコンパイルできなければならないことを意味します。

``nvcc`` を使用する ``CUDA`` バックエンドでは、これは構成時に ``CMAKE_CXX_COMPILER=nvcc`` を設定することを意味します。しかし、``nvcc`` はホストコンパイルとデバイスコンパイルで別々のフラグを必要とします。
``nvcc`` の別々のフラグに関するこの要件は、同じターゲットにリンクされる他のライブラリもこれに従う必要があることを意味します。
``clang`` などの他のコンパイラは、ホストフラグとデバイスフラグを区別する必要がないため、ラッパースクリプトを必要とせず、``CMAKE_CXX_COMPILER`` を介して直接使用できます。

この問題を解決するために、Kokkos には ``bin`` サブディレクトリにある ``nvcc_wrapper`` という小さな ``bash`` スクリプトが付属しています。
このスクリプトには2つの機能があります。コンパイルおよびリンクコマンドを ``nvcc`` にリダイレクトすること、そして与えられたコンパイラおよびリンカフラグをホストコンパイラ用とデバイスコンパイラ用に振り分けることです。

これを明示的に使用するには、``CMAKE_CXX_COMPILER`` を適切に設定します。コンパイラがデバイスコードをコンパイルできない場合、Kokkos は ``kokkos_launch_compiler``（次のセクションを参照）を使用して呼び出しを ``nvcc_wrapper`` に自動的にリダイレクトします。

kokkos_launch_compiler: 何をするのか、どう制御するのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

複数のライブラリに依存する複雑なソフトウェアプロジェクトでは、有効化されたバックエンドのために Kokkos が必要とするコンパイラが、プロジェクトが依存する他のライブラリの1つをコンパイルできない場合があります。
このような状況を回避するために、Kokkos は ``kokkos_launch_compiler`` という別の ``bash`` スクリプトを導入しました。
このスクリプトは、Kokkos を使用する ``C++`` ファイルをコンパイルするコンパイラおよびリンカコマンド**のみ**を、Kokkos コードをコンパイルできるコンパイラ（例: ``CUDA`` バックエンドの ``nvcc_wrapper``）にリダイレクトします。Kokkos を使用しない ``C++`` ファイルのコンパイラおよびリンカコマンド、または異なる言語のファイルはリダイレクトされません。

``bin`` サブディレクトリにあるこのスクリプトは、``CMake`` のコンパイラランチャーのように使用することを意図しています。
しかし（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で構成される場合を除く）、Kokkos は ``CMake`` が使用する ``CXX`` コンパイラが有効化されたバックエンドのコードをコンパイルできるかどうかを検出しようとします。``CXX`` コンパイラがバックエンドコードをコンパイル**できない**場合、Kokkos は自動的に ``kokkos_launch_compiler`` を使用します。それが ``nvcc_wrapper`` にリダイレクトする場合、``nvcc`` のホストコンパイラを ``CXX`` コンパイラに設定します。この仕組みの目的は、Kokkos のパフォーマンスポータブルなビルドシステムの利用を簡素化することです。

これはほとんどのユースケースをカバーしますが、Kokkos はユーザーが ``kokkos_launch_compiler`` を**常に**または**決して**使用しないように要求する方法を提供しています。
``kokkos_launch_compiler`` を常に使用するには、ユーザーは ``find_package`` を呼び出す際に ``launch_compiler`` コンポーネントを要求できます:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS launch_compiler)

対照的に、``kokkos_launch_compiler`` を決して使用したくない場合:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS separable_compilation)

バックエンドの CMake 言語でコンパイルする
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

対応する専用の ``CMake`` 言語を持つ ``CUDA`` および ``HIP`` バックエンドでは、Kokkos がそれぞれの ``CMake`` 言語でのみコンパイラおよびリンカフラグを設定するように構成できます。これにより、Kokkos は事実上 ``CMake`` に対して ``CUDA`` または ``HIP`` ライブラリとして識別されます（``CXX`` ライブラリとして識別される**代わりに**）。これにより追加のホストバックエンドを有効化できます（例: ``CUDA``、``OPENMP``、``SERIAL`` を ``CUDA`` 言語モードでコンパイルできます）。
これは、Kokkos にリンクするファイルもそれぞれの ``CMake`` 言語として識別される必要があり、Kokkos で有効化されているアーキテクチャ向けにコンパイルされる必要があることを意味します。
この動作を有効にするには、オプション `Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE <configuration-guide.html#backend-specific-options>`_ を有効にします。詳細な例と説明については、そのオプションに関する注記を確認してください。

複数の言語でコンパイルする
^^^^^^^^^^^^^^^^^^^^^^^^^^

複雑なソフトウェアプロジェクトでは、Kokkos を使用する複数のライブラリを使用する場合があります。これらの1つがバックエンドの ``CMake`` 言語（例: ``Cuda`` または ``HIP``）で Kokkos を使用し、他のライブラリが Kokkos を ``CXX`` ライブラリとして使用する場合、両方のモードでコンパイルできる Kokkos インストールが必要になります。
これを可能にするために、Kokkos はオプション `Kokkos_ENABLE_MULTIPLE_CMAKE_LANGUAGES <configuration-guide.html#backend-specific-options>`_ を提供しています。詳細と要件については、そのオプションに関する注記を確認してください。
