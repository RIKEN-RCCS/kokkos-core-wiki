高度な設定とビルド
==================

nvcc_wrapper: なぜ必要なのか、そして何をするのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

一般に（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で設定されている場合を除く）、Kokkos に依存する性能可搬なコードは ``CMake`` において自身を ``CXX`` コードとして識別します。
これは、``CMake`` が ``CXX`` コードのコンパイルに使用するコンパイラーが、Kokkos で有効化されたバックエンドおよびアーキテクチャー向けにコードをコンパイルできなければならないことを意味します。

``nvcc`` を使用する ``CUDA`` バックエンドの場合、これは設定時に ``CMAKE_CXX_COMPILER=nvcc`` を設定することを意味します。しかし ``nvcc`` はホストコンパイルとデバイスコンパイルで別々のフラグを必要とします。
``nvcc`` が別々のフラグを必要とするというこの要件は、同じターゲットにリンクされる他のライブラリーもこれに従う必要があることを意味します。
``clang`` などの他のコンパイラーはホストとデバイスのフラグを区別する必要がないため、ラッパースクリプトを必要とせず、``CMAKE_CXX_COMPILER`` を介して直接使用できます。

この問題でユーザーを助けるために、Kokkos には ``bin`` サブディレクトリーにある ``nvcc_wrapper`` という小さな ``bash`` スクリプトが付属しています。
このスクリプトには 2 つの機能があります。コンパイルおよびリンクコマンドを ``nvcc`` にリダイレクトすることと、与えられたコンパイラーおよびリンカーフラグをホスト用とデバイスコンパイラー用に振り分けることです。

これを明示的に使用するには、``CMAKE_CXX_COMPILER`` を適切に設定します。コンパイラーがデバイスコードをコンパイルできない場合、Kokkos は ``kokkos_launch_compiler``（次のセクションを参照）を使用して、呼び出しを ``nvcc_wrapper`` に自動的にリダイレクトします。

kokkos_launch_compiler: 何をするのか、そしてどのように制御するのか
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

複数のライブラリーに依存する複雑なソフトウェアプロジェクトでは、有効化されたバックエンドに Kokkos が必要とするコンパイラーが、プロジェクトが依存する他のライブラリーの一つをコンパイルできない場合があります。
このような状況を回避するために、Kokkos は ``kokkos_launch_compiler`` という別の ``bash`` スクリプトを導入しました。
このスクリプトは、Kokkos を使用する ``C++`` ファイルをコンパイルするコンパイラーおよびリンカーコマンド**のみ**を、Kokkos コードをコンパイルできるコンパイラー（例えば ``CUDA`` バックエンド向けの ``nvcc_wrapper``）にリダイレクトします。Kokkos を使用しない ``C++`` ファイルや、異なる言語のファイルのコンパイラーおよびリンカーコマンドはリダイレクトされません。

このスクリプトは ``bin`` サブディレクトリーにあり、``CMake`` におけるコンパイラーランチャーのように使用することを意図しています。
しかし（``Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE=ON`` で設定されている場合を除く）、Kokkos は ``CMake`` が使用する ``CXX`` コンパイラーが有効化されたバックエンド向けのコードをコンパイルできるかどうかを検出しようとします。``CXX`` コンパイラーがバックエンドコードをコンパイルできない場合、Kokkos は自動的に ``kokkos_launch_compiler`` を使用します。``nvcc_wrapper`` にリダイレクトする場合、``nvcc`` のホストコンパイラーを ``CXX`` コンパイラーに設定します。この考え方は、Kokkos の性能可搬なビルドシステムの使用を簡素化することです。

これはほとんどのユースケースをカバーしていますが、Kokkos はユーザーが ``kokkos_launch_compiler`` を**常に**または**決して**使用しないように要求する方法を提供しています。
``kokkos_launch_compiler`` を常に使用するには、ユーザーは ``find_package`` を呼び出す際に ``launch_compiler`` コンポーネントを要求できます:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS launch_compiler)

対照的に、``kokkos_launch_compiler`` を決して使用したくない場合:

.. code-block:: sh

   find_package(Kokkos REQUIRED COMPONENTS separable_compilation)

バックエンドの CMake 言語でのコンパイル
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

対応する専用の ``CMake`` 言語を持つ ``CUDA`` および ``HIP`` バックエンドの場合、Kokkos はそれぞれの ``CMake`` 言語でのみコンパイラーおよびリンカーフラグを設定するように構成できます。これにより Kokkos は事実上、``CMake`` に対して ``CUDA`` または ``HIP`` ライブラリーとして識別されます（``CXX`` ライブラリーとして識別される**代わりに**）。これにより追加のホストバックエンドを有効化できます（例えば ``CUDA``、``OPENMP``、``SERIAL`` は ``CUDA`` 言語モードでコンパイルできます）。
これは、Kokkos にリンクするファイルもそれぞれの ``CMake`` 言語として識別される必要があり、Kokkos で有効化されたアーキテクチャー向けにコンパイルされる必要があることを意味します。
この動作を有効にするには、オプション `Kokkos_ENABLE_COMPILE_AS_CMAKE_LANGUAGE <configuration-guide.html#backend-specific-options>`_ を有効にします。詳細な例と説明については、このオプションに関する注記を確認してください。

複数の言語でのコンパイル
^^^^^^^^^^^^^^^^^^^^^^^^

複雑なソフトウェアプロジェクトでは、Kokkos を使用する複数のライブラリーを使用する場合があります。それらの一つがバックエンドの ``CMake`` 言語（例えば ``Cuda`` や ``HIP``）で Kokkos を使用し、他のライブラリーが Kokkos を ``CXX`` ライブラリーとして使用する場合、両方のモードでコンパイルできる Kokkos インストールが必要です。
これを可能にするために、Kokkos はオプション `Kokkos_ENABLE_MULTIPLE_CMAKE_LANGUAGES <configuration-guide.html#backend-specific-options>`_ を提供しています。詳細と要件については、このオプションに関する注記を確認してください。
